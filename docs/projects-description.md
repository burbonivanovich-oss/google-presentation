# Проект A. Отчётность и прогноз продаж

## 1. Цель задачи и предпосылки

### Откуда взялась задача
Квартальная отчётность по продажам Контур.Маркета (направления Розница, Общепит, Кассы, ОФД, продукт Маркет, контекстная реклама) собиралась вручную: аналитик выгружал несколько крупных xlsx («Царь свод», «Царь продажи», выгрузки рекламных кампаний), сводил их в таблицы, считал QoQ/YoY и переносил цифры в презентацию. На каждый отчётный цикл уходило несколько дней, цифры приходилось пересобирать заново, а ошибки переноса всплывали уже в готовых слайдах.

Параллельно от руководства появился запрос на **прогноз выполнения годового плана 2026** по ключевым продуктам (Маркет и ОФД) — не «на глазок», а на основе моделей и с прозрачной методологией.

### Проблема
Ключевые проблемы:
- **Ручная сборка медленная и хрупкая.** Источники — это сводные xlsx «по всем разрезам» с фильтрами в первой строке и разной структурой; каждый разрез приходилось вычищать вручную.
- **Нет единого источника правды.** Одни и те же продукты в разных выгрузках размечены по-разному: по «Бизнес-юниту», по «Проекту», по «Продукту» — без согласованного маппинга цифры расходятся (например, Маркет как «Продукт» = 119 млн, а как «Проект общий» = 70 млн).
- **Прогноз отсутствовал как процесс.** Не было ни моделей, ни сценариев, ни методологии — только экстраполяция «в уме».

### Решение
Создан Python-пакет `reportgen`, который закрывает весь цикл:

1. **Агрегация.** Читает исходные xlsx прямо из Google Drive, чистит (находит реальную шапку, парсит даты и числа с пробелами/валютой), группирует по направлениям через согласованный маппинг.
2. **Сводка.** Собирает Google-таблицу из 10 листов (обзор, направления, тарифы, реклама, источники) — единый выверенный источник для презентации.
3. **Презентация.** Генерирует HTML-дек в фирменном стиле Контур (палитра Blue из `theme1.xml`, шрифт Montserrat) с SVG-графиками, разделами и слайдами прогноза.
4. **Прогноз.** Считает выполнение годового плана 2026 ансамблем из четырёх моделей и выгружает Excel с методологией.
5. **Автоматизация.** Запуск через GitHub Actions (workflow_dispatch) — без локального окружения.

## 2. Хронология работы

**Старт.** Спроектирована схема сводки (10 листов) и архитектура: `aggregator.py` (чтение xlsx + группировки), `aggregator_writer.py` (запись Google Sheet), CLI-команда `aggregate`.

**Итерация по маппингу направлений.** Первый прогон дал нули по Рознице/Общепиту/Кассам — фильтр по «Сегментный тариф» оказался неверным. По диагностике (`inspect-sales`) перешли на «Бизнес-юнит»: Розница = Госсистемы для розницы, Кассы = Кассовики, Общепит = Госсистемы для общепита, ОФД — продуктовый срез по тарифу.

**Презентация в HTML.** Чтобы крутить дизайн без прогона облака, собран самодостаточный HTML-дек. Первая версия была в оранжевом — после сверки с брендшаблоном «Контур Blue 2023 Montserrat» цвета заменены на фирменные (синий `#2291FF`), шрифт — Montserrat, вёрстка — масштабирование под весь экран, CSS-бары заменены на SVG-графики.

**Прогноз 2026.** Реализован модуль моделей (`forecast_models.py`) и Excel-выгрузка (`forecast_excel.py`). Ключевая итерация — определение охвата: план размечен по проектам и по продуктам, значения расходятся. После сверки с дашбордами зафиксированы охваты: **ОФД = проект п453** (37,5 млн), **Маркет = «Продукт = Маркет»** (119 млн); выверенные помесячные ряды сверены с итоговой строкой дашборда до рубля.

## 3. Архитектура и методология прогноза

### Источники и маппинг
- **Царь свод все разрезы** — помесячный план/факт.
- **Царь продажи** — транзакции 2025–2026 (для разрезов по направлениям и тарифам).
- **data (15–17)** — рекламный факт (целевые лиды) по продуктам.
- **Дашборды Контекст** — расход, CPL, CPO, оплаты (сняты с экрана, суммы сверены с «Итого»).
- **План-факт по продуктам** (data 22/25/26) — для прогноза: ОФД = п453, Маркет = Продукт=Маркет.

### Модели прогноза (ансамбль)
Горизонт: факт по последний завершённый месяц (май), прогноз июнь–декабрь. Обучающий ряд ~17 месяцев.

| Модель | Суть |
|---|---|
| **Plan-pacing** | остаток = план месяца × (факт YTD ÷ план YTD) |
| **Сезонный YoY** | месяц = факт того же месяца 2025 × (1 + прирост YTD) |
| **ETS / Holt** | экспоненциальное сглаживание с трендом (`statsmodels`) |
| **Регрессия + сезонность** | линейный тренд + месячные дамми (`scikit-learn`) |

База = взвешенное среднее (0.40 / 0.25 / 0.15 / 0.20). Коридор низ/верх = min/max по методам помесячно. Отрицательные значения обнуляются.

| Продукт | Годовой план | Факт YTD (5 мес) | YoY | Прогноз года (база) | % плана (низ/база/верх) | Вывод |
|---|---|---|---|---|---|---|
| **Маркет** | 119,0 млн | 44,5 млн (91%) | −21% | 102,0 млн | 78 / **86** / 91% | недовыполнение |
| **ОФД** | 37,5 млн | 18,9 млн (123%) | −16% | 43,3 млн | 103 / **115** / 124% | перевыполнение |

## 4. Выводы и следующие шаги

### Ключевые выводы
- Полный цикл «xlsx → сводка → презентация/прогноз» автоматизирован и воспроизводим через GitHub Actions; сборка занимает минуты вместо дней.
- Маппинг продуктов согласован и зафиксирован в коде, расхождения охватов задокументированы.
- Прогноз показывает: **Маркет идёт к ~86% плана** (падение −21% YoY — главный риск), **ОФД перевыполняет (~115%)**.

# Проект B. Бот мониторинга (Wordstat + Telegram)

## 1. Цель задачи и предпосылки

### Откуда взялась задача
Команде нужно было оперативно отслеживать **поисковый спрос** (Yandex Wordstat) и **упоминания в инфополе** — в крупных Telegram-каналах СМИ (РИА Новости, ТАСС, РБК и др.). Ручной просмотр десятка каналов и выгрузка Wordstat вручную не масштабируются и не дают регулярности.

### Проблема
- Нет единой точки сбора: спрос — в одном кабинете, упоминания — в десятке разных каналов.
- Нет регулярности: важные упоминания всплывают постфактум.
- Ручная работа не воспроизводима и съедает время аналитика.

### Решение
Python-бот, который:
- тянет топ-запросы и аналитику из **Yandex Cloud Search API v2 (Wordstat)**;
- мониторит **12 Telegram-каналов** на вхождение ключевого слова **«тс пиот»** через Telethon;
- формирует и отправляет **отчёты по расписанию** (понедельник, вторник).

### Артефакты

| Описание | Артефакт |
|---|---|
| Репозиторий бота | `github.com/burbonivanovich-oss/Test` |
| Основной скрипт | `wordstat_bot.py` |
| Конфигурация | `config.yaml` (каналы, ключевые слова, расписание) |
| Авторизация Telegram | `setup_session_auto.py`, `setup_session_qr.py` |
| Модуль мониторинга каналов | `telegram_channel_monitor/` (парсинг, фильтрация, форматирование) |
| Документация | `README.md`, `QUICK_START.md`, `SETUP_GUIDE.md` |

## 2. Архитектура и стек

- **Язык:** Python 3.11+.
- **Библиотеки:** Telethon (Telegram), requests (HTTP/API), PyYAML (конфиг).
- **Внешний API:** Yandex Cloud Search API v2 (миграция с устаревшего Wordstat API описана в README).
- **Развёртывание:** инструкция под bothost.ru; сессия Telegram создаётся через авторизацию по коду или QR.
- **Конфигурация в `config.yaml`:** список каналов, ключевые слова, расписание рассылки.

**Поток работы:** по расписанию → запрос Wordstat (топ-запросы, динамика) + проход по каналам (новые сообщения → фильтр по ключевым словам) → агрегация → форматирование отчёта → отправка в Telegram.

## 3. Выводы и следующие шаги

### Ключевые выводы
- Мониторинг спроса и инфополя сведён в один автоматический процесс с регулярной рассылкой.
- Конфигурация (каналы, ключевые слова, расписание) вынесена в `config.yaml` — расширяется без правки кода.
